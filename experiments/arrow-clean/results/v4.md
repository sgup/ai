## P1 — Diagnose: reproduce and find the true root cause

### Step 1 — Reproduce the exact symptom (confirmed)

The report says: as of Jan 9 2026, an event ~15–16 days out reads "in/a month" instead of "2 weeks". I reproduced it directly:

```

now = arrow.Arrow(2026, 1, 9)
for d in [13,14,15,16,17,20,21,22,28]:
    later = now.shift(days=d)

```

Confirmed output (`later.humanize(now)` is the user-facing "in X" framing):

```

14 -> in 2 weeks
15 -> in a month   <-- bug starts here
16 -> in a month

```

So the symptom is real and the boundary is sharp: **14 days = "in 2 weeks", 15 days = "in a month".** Everything from 15 days up to a real calendar month collapses to "a month". (The same thresholds drive the "ago" direction; `now.humanize(later)` over +8..+45d gives the mirror result — confirmed in a separate run.)

### Step 2 — Read the code path that produced it (confirmed)

`humanize()` is defined at `arrow/arrow.py:1133`. The relevant logic is the `granularity == "auto"` branch, `arrow/arrow.py:1191-1262`. After the sub-day branches (1192-1215), it computes a calendar diff and a month bump:

```
1217  calendar_diff = (relativedelta(dt, self._datetime) if self._datetime < dt
1218                   else relativedelta(self._datetime, dt))
1222  calendar_months = calendar_diff.years * self._MONTHS_PER_YEAR + calendar_diff.months
1226  # For months, if more than 2 weeks, count as a full month
1227  if calendar_diff.days > 14:
1228      calendar_months += 1
1230  calendar_months = min(calendar_months, self._MONTHS_PER_YEAR)
```

Then the unit is selected by a chain of `elif`s in this order (1232-1262): `day` (<2d) → `days` (<1wk) → **`month`/`months` if `calendar_months >= 1` and `diff < 1yr`** (1239) → `week` (<2wk) (1250) → `weeks` (<1mo) (1253) → `year`/`years`.

The ordering is the crux: the **months branch at line 1239 sits ahead of the week/weeks branches at 1250-1255.** So once `calendar_months >= 1`, the function returns a "month" answer and the week branches are never reached.

### Step 3 — Confirm the mechanism, not just a plausible story (confirmed)

I replicated the internal computation for the boundary inputs to see exactly which line fires:

```
PYTHONPATH=/tmp/arrow-clean/arrow python3 -c "
import arrow
from dateutil.relativedelta import relativedelta
now = arrow.Arrow(2026, 1, 9)
for d in [13,14,15,16,17,20]:
    later = now.shift(days=d)
    cd = relativedelta(later._datetime, now._datetime)
    base = cd.years*12 + cd.months
    bumped = base + (1 if cd.days > 14 else 0)
    print(d, 'rel.days=',cd.days, 'rel.months=',cd.months,
          'calendar_months base=',base,'bumped=',bumped, '->', later.humanize(now))"
```

Confirmed:

```
13 rel.days=13 rel.months=0  base=0 bumped=0 -> in a week
14 rel.days=14 rel.months=0  base=0 bumped=0 -> in 2 weeks
15 rel.days=15 rel.months=0  base=0 bumped=1 -> in a month
16 rel.days=16 rel.months=0  base=0 bumped=1 -> in a month
17 rel.days=17 rel.months=0  base=0 bumped=1 -> in a month
20 rel.days=20 rel.months=0  base=0 bumped=1 -> in a month
```

**True root cause (confirmed):** for 15–~27 day gaps, `relativedelta` correctly reports `months=0, days=15..27`. Line 1227 (`calendar_diff.days > 14`) then bumps `calendar_months` from 0 to 1. Because the `calendar_months >= 1` branch (line 1239) is ordered *before* the week branches (1250-1255), the function returns `"a month"` and never evaluates "is this < 2 weeks / < a month → weeks". The `> 14` bump is a half-week off and, combined with branch ordering, swallows the entire "2–4 weeks" range.

### Step 4 — Rule out the plausible-but-wrong alternative (confirmed)

The tempting wrong answer is "`_SECS_PER_MONTH = 30.5 days` (arrow/arrow.py:131) is miscalibrated." That constant is **never reached** for these inputs — the months branch (1239) short-circuits before the `weeks` branch (1253) that uses `_SECS_PER_MONTH`. I confirmed the boundary is purely `relativedelta.days`-driven and independent of `_SECS_PER_MONTH` / the specific month by scanning multiple base dates:

```
2026-01-09: 14d:in 2 weeks | 15d:in a month | 16d:in a month
2026-02-01: 14d:in 2 weeks | 15d:in a month | 16d:in a month
2026-02-15: 14d:in 2 weeks | 15d:in a month | 16d:in a month
2026-01-20: 14d:in 2 weeks | 15d:in a month | 16d:in a month
```

Same 14/15 cutover regardless of base date. So the cause is the `days > 14` bump + branch ordering, **not** `_SECS_PER_MONTH`. (The comment at line 1226, "if more than 2 weeks, count as a full month," names the original — and in my view flawed — intent: it treats anything past 14 days as a month, which is exactly the over-eager rounding the user is reporting.)

## P2 — Fix design: where, how, and how to confirm correct AND complete

### Where

A single region: `arrow/arrow.py`, the "auto" branch, lines **1226-1230** (the `days > 14` bump) in relation to the branch ordering at **1239-1255**. No other call site or locale file needs to change — the locale `describe()` already renders `week`/`weeks`/`month`/`months` correctly; the defect is purely threshold/ordering selection logic upstream of it.

### How (design, not implemented)

The intent we want: 15–~25 days should read in **weeks**, and "a month" should begin only near a true calendar month (~26–31 days, i.e. when `relativedelta.months` is genuinely 1, or the day remainder is close to a full month). Two viable shapes, with my recommendation first:

1. **Recommended — make the bump honest and reorder so weeks win in the gap.** Only promote to a month when the remainder is genuinely near a month, not merely past two weeks: e.g. require `calendar_diff.days > 25` (close to a full month) for the `+= 1` bump *when `months == 0`*, and ensure the week/weeks branches (1250-1255) are consulted before the month branch when `calendar_months` was produced purely by the day-remainder bump. This keeps the "round up near a month" behavior the comment intended while letting 15–24 days fall through to "weeks". I'd treat the exact cutover (25 vs 26 vs "only bump when `days >= ~28`") as a deliberate product choice and surface it, not silently pick.
2. **Alternative — drop the day-remainder bump entirely** and let `calendar_months` reflect only real `relativedelta.months`, relying on the existing `weeks` branch (`diff < _SECS_PER_MONTH`, line 1253) to cover everything up to ~30.5 days. Simpler, but it changes the boundary where "a month" first appears (≈30.5d instead of ~26d) and may read as a behavior change for the 26–30 day range — so it needs the same product sign-off.

Both are low-blast and reversible (revert the diff). I would **not** touch `_SECS_PER_MONTH`; it is not the cause and changing it would shift unrelated boundaries.
