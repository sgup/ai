## P1 — Coupon under-charge

### Order of investigation

1. **Reproduce against the spec, not a hunch.** The spec (`README.md` lines 7-8) says coupons are *capped at 50% — a larger coupon is clamped to 50%*. The reported symptom is "charged less than they should have." The first question: does the code enforce that cap? I check `apply_coupon` in `src/pricing.py:22-24` and the one place it's called for the web charge, `order_total` at `src/pricing.py:35-39`.

2. **Trace the call chain rather than trust the docstring.** `apply_coupon`'s docstring says "percent_off: 0-100" but says nothing about clamping; `order_total` passes `percent_off` straight through with no clamp. So the clamp the spec requires lives in **neither** function. I confirmed the consequence by running it:
   - `apply_coupon(100, 80)` → `20.0` **[confirmed]**
   - `order_total([{price:100, qty:1}], 80)` → `21.6` **[confirmed]**
   - With the spec's 50% cap, an 80% coupon should clamp to 50%: subtotal 100 → 50 after coupon → ×1.08 tax → **54.0**. The code charges **21.6**. That under-charge of ~$32 on a $100 order **is** the bug. **Root cause [confirmed]: the 50% coupon cap is unimplemented.**

3. **Rule out the other under-charge paths before declaring the cause complete** (don't promote a single sample to "the cause" — there may be more than one leak):
   - **Negative `percent_off`** would *increase* the charge, not decrease it — wrong direction for this report, so not the customer's symptom, but still a validity hole to note. **[inferred — would confirm with `apply_coupon(100, -10)`]**
   - **`percent_off > 100`** produces a *negative* amount (a credit). Worse than under-charge. The same missing-clamp fix on the upper bound covers it. **[inferred from the formula; confirm by running `apply_coupon(100, 150)`]**
   - **Where does `percent_off` enter?** The spec (line 12) claims prices are validated upstream, but says nothing about the *coupon* being validated upstream. I'd confirm by reading the actual callers of `order_total`/`apply_coupon` — neither `src/` file clamps, so if no caller does either, untrusted coupon values reach the math raw. I did **not** find a caller in this repo that clamps. **[confirmed within this repo; the true upstream is outside it — I'd name that as the boundary to check, not assume]**

4. **Decide the fix location (name the fork, even having chosen).** Two candidates:
   - (a) Clamp inside `apply_coupon` — one function, both entry points (`order_total` and `checkout.charge_amount`, which both call it) inherit the cap automatically. **Recommended**: single source of truth, and the spec frames the cap as a property of the coupon itself.
   - (b) Clamp in each caller — rejected: duplicates the rule in two files and invites them to drift, which is exactly the class of bug P2 is about.
   I'd note this choice but **not implement** it here.

### How I'd confirm the fix is right *and complete*

- **A failing test first.** Add a test that pins the spec number — `order_total([{price:100,qty:1}], 80) == 54.0` — and watch it **fail on today's code** (it returns 21.6). A fix is only proven by a test that was red before and green after. The existing `is not None` test cannot catch this.
- **Boundary table**: `percent_off` ∈ {0, 50, 51, 80, 100, 150, -10} checked against hand-computed spec values, including the clamp edges at exactly 50 and just above.
- **Both entry points**: assert the same cart+coupon yields consistent treatment through `order_total` and `charge_amount` (the spec's line 14 "must always agree" — see P2; this is where the clamp fix must not stop at one file).
- **Re-run the whole gate and report the delta**, not a grep of my own change: "baseline 2 passed → now N passed, +the new clamp tests," read from a real exit code.
- **The one claim I'd most expect to be wrong**: that fixing only `apply_coupon` makes the system correct. It removes the under-charge, but the two entry points still disagree on tax (P1's cart, run below). I would not call P1 "complete" without saying that out loud.

## P2 — Ship review

I'd review the module against its own spec (`README.md`) line by line, since the spec is the source of truth and is small enough to check exhaustively. Each rule, the code that implements it, and what I found:

1. **Subtotal** (spec line 6) — `subtotal` (`pricing.py:6-11`) sums `price*qty`. Matches the rule. Uses float arithmetic on money (see #5). **[confirmed correct against spec]**

2. **Coupon cap at 50%** (spec lines 7-8) — **unimplemented.** Same defect as P1. `order_total([...], 80)` → `21.6` instead of `54.0`. **[confirmed]** Blocker.

3. **Two entry points must agree** (spec line 14) — **they do not.** I ran the same cart through both:
   - `order_total([{price:100,qty:1}], 50)` → **54.0** (applies tax) **[confirmed]**
   - `charge_amount({items:[...]}, 50)` → **50.0** (no tax) **[confirmed]**
   `charge_amount` (`checkout.py:6-9`) calls `subtotal` + `apply_coupon` but **never applies the 8% tax** that `order_total` does and that spec line 9 says *every charge includes*. The mobile API under-charges every taxed order by the full tax amount. **[confirmed]** This is a second, independent money leak and a harder blocker than the coupon — it's silent and affects *every* mobile order, not just couponed ones.

4. **Valid cart = all items `price > 0` and `qty > 0`** (spec line 11) — `is_valid_cart` (`pricing.py:27-32`) is wrong on **two** axes:
   - It uses `>= 0`, so a `price: 0` / `qty: 0` item passes. Run: `is_valid_cart([{price:0, qty:0}])` → `True` **[confirmed]** (spec wants `> 0`, i.e. `False`).
   - It returns `True` on the **first** item that passes — that's **ANY**, not **ALL**. Run: `is_valid_cart([{price:-5, qty:1}, {price:5, qty:1}])` → `True` **[confirmed]** (a cart with a negative-price item is accepted because a later item is fine).
   This is a check that effectively cannot fire for its intended purpose — I'd name it as broken, not as a "convention." Blocker (free / negative-priced items pass validation). **[confirmed]**

5. **Money in floats** — `subtotal`, `apply_coupon`, `parse_price` all use `float`. Float rounding can shift a cent on real carts; the spec rounds only at the boundary. I'd flag this as a correctness *risk* for a payments module (the standard remedy is integer cents or `Decimal`), but I'd scope it as a follow-up recommendation, **not** block the week's ship on it unless review wants it in — it's a larger change than the three blockers above. **[inferred risk — would confirm by constructing a cart whose float sum mis-rounds; flagged honestly either way]**

6. **`parse_price` swallows bad input to `0.0`** (`pricing.py:14-19`) — returns `0.0` on `ValueError`/`TypeError`. The spec (line 12-13) claims prices are normalized upstream so this never receives malformed input — but if that contract is ever violated, this **silently zeroes a real price** instead of failing loudly, which on a payments path means charging $0. I'd name that as a latent flaw (a default that silently zeroes a real measurement), confirm whether any caller actually relies on it, and recommend it raise rather than swallow. **[confirmed behavior by reading; the "never reached" claim is upstream and outside this repo — I'd verify it, not assume it]**

### How I'd decide it's safe to deploy

- **Three correctness blockers must be fixed first**, each with a spec-pinned regression test that is red before and green after: (a) coupon 50% clamp, (b) `charge_amount` missing tax, (c) `is_valid_cart` `>=`-and-ANY → `>`-and-ALL. These are the ones that move money or admit bad carts.
- **Cross-entry-point parity test** as the gate for the "must agree" rule (spec line 14): a property-style test that runs a range of carts and coupons through *both* `order_total` and `charge_amount` and asserts they reconcile, so they can't drift again. This is the single highest-value test the suite lacks today.
- **Fix the test gate itself.** The current `is not None` assertion is not a correctness gate. I'd replace/augment it with value assertions, and **resolve the pytest gap** (install it or update the README to the command that actually runs) rather than ship on a README command that doesn't execute in this env. Reported as a blocker-to-CI, not worked around.
- **Re-run the full suite, read the real exit code, report the delta vs the 2-passed baseline**, and name what only a human can confirm: that "validated upstream" (spec line 12) is actually true in the live callers — that boundary lives outside this repo and I can't confirm it from here.
- **Decision rule**: ship only when all three money/validity blockers have a red→green test, the parity test passes, and the float-money + `parse_price`-zeroing risks are explicitly accepted-or-deferred in writing (not silently). I would **not** sign off this week on the float-money issue being *fixed* — only on it being *known and consciously deferred*.
