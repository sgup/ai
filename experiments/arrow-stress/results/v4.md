### 3. The true root cause (confirmed by tracing, not by name)
Two independent defects compound, and I confirmed each by instrumenting the actual values:

**(a) Branch-ordering bug — the week branches are dead code for 15–27 days.** The `calendar_months >= 1` test at line **1239** sits *before* the "a week"/"N weeks" branches at lines **1250 and 1253**. Because `elif` short-circuits, once `calendar_months >= 1` is true the week branches can never be reached.

**(b) The `days > 14` inflation manufactures a phantom month from a pure sub-month day count.** For a 15-to-27-day diff, `relativedelta` yields `months=0` with `days` in 15..27, so `calendar_months` starts at **0**. Line 1227's `calendar_diff.days > 14` is then true, so line 1228 bumps it to **1**. Confirmed by direct instrumentation:
```
d=15: relativedelta(months=0,days=15) calendar_months=0 -> inflated=1
d=20: relativedelta(months=0,days=20) calendar_months=0 -> inflated=1
d=27: relativedelta(months=0,days=27) calendar_months=0 -> inflated=1
```
So the code says "1 month" for something that is, in calendar terms, **zero months and ~2–4 weeks**.

Putting (a)+(b) together: for any diff of 15–27 days, the inflated `calendar_months == 1` makes the `elif` at 1239 fire and return "a month", and the week branches at 1250/1253 are unreachable. **That is the actual cause.**

Note the `days > 14` inflation is *not* wholly wrong — it has a legitimate second role I confirmed: for genuine month-plus-remainder gaps it rounds up correctly (`1mo+20d` → calendar_months 1, days=20>14 → 2 → "2 months"; `45 days` → months=1,days=14, not >14 → stays "a month"). The flaw is that it also fires when `calendar_months` is **0**, fabricating a month that doesn't exist. So the fix must preserve the round-up-real-months behavior while suppressing the phantom-month case — naming this honestly matters because a naive "delete the +1" would regress the legitimate case.

### Where
Inside the `granularity == "auto"` block of `humanize()`, `arrow/arrow.py:1191–1262`. Two coordinated edits:

1. **Stop the phantom month (line 1226–1228).** Only round up to a month when there is already at least one whole calendar month present — i.e., gate the `+1` so it cannot turn 0 months into 1. Conceptually: `if calendar_months >= 1 and calendar_diff.days > 14: calendar_months += 1` (round up a *real* month's remainder), and do **not** invent a month when `calendar_months == 0`.
